# **AI部署场景性能评估方案**

## **摘要**

本方案旨在为在单BPU核心的AI开发板上部署两个独立AI任务，并计划未来扩展非AI业务的场景，提供一套完整的性能评估与资源规划框架。方案将从BPU、CPU、内存和带宽四个维度出发，明确核心考量方向，提供可量化的具体分析步骤，并最终形成一个用于评估未来业务资源可用性的决策框架。

---

# **1. 硬件资源占用的核心考量方向**

在进行量化分析之前，我们必须首先确定需要关注的维度和潜在的瓶颈。

## **1.1. BPU核心 (Brain Processing Unit)**

由于BPU是唯一的、专门的AI计算资源，其利用和调度是性能分析的核心。

- **计算饱和度 (Compute Saturation):** BPU是否已经满负荷运行？是否存在空闲周期？这是评估能否承载更多AI任务（如果未来有此计划）或其空闲周期能否被其他系统利用的基础。
    
- **任务调度与并发执行 (Task Scheduling & Concurrency):** 只有一个核心，两个模型必须分时复用。需要分析：
    
    - **调度开销 (Scheduling Overhead):** 切换两个模型任务（上下文切换）所带来的时间开销。
        
    - **执行模式 (Execution Model):** 两个任务是严格的轮流执行（乒乓模式），还是基于某种优先级进行抢占式调度？这会影响每个任务的实时性和平均延迟。
        
- **计算延迟 (Inference Latency):** 单个模型完成一次推理所需的时间。这直接决定了每个任务能达到的最高帧率（FPS）。
    
- **功耗与散热 (Power & Thermal):** BPU高负载会带来功耗和温度的上升，这可能导致硬件降频（Thermal Throttling），从而影响性能的稳定性。

## **1.2. CPU核心 (Central Processing Unit)**

CPU负责除模型推理外的所有计算，包括系统运行、任务调度、数据处理和未来的业务逻辑。

- **总体与核心间负载均衡 (Overall & Per-Core Load Balancing):**
    
    - 总CPU利用率是多少？
        
    - 如果是多核CPU，负载是否均匀分布？是否存在某个核心被打满，成为性能瓶颈？
        
- **任务归属分析 (Workload Attribution):** 必须清晰地将CPU的开销分解到不同的任务上：
    
    - **OS与系统服务开销:** 操作系统自身运行所需的基础资源。
        
    - **AI任务前后处理开销:** 分别归属于任务1和任务2的数据预处理（如resize, normalize）和后处理（如解析输出、非极大值抑制NMS）的CPU占用。
        
    - **BPU驱动与任务管理开销:** CPU用于提交BPU任务、查询状态和获取结果的开销。
        
    - **空闲时间 (Idle Time):** 这是未来业务代码可利用的直接资源。
        
- **中断与上下文切换 (Interrupts & Context Switching):** 过高的中断请求（IRQs）或进程/线程上下文切换频率会消耗大量CPU周期，即使CPU利用率看起来不高，系统响应也可能很慢。

## **1.3. 内存/RAM (Random Access Memory)**

内存是数据流转和状态保存的关键，其容量和带宽都可能成为瓶颈。

- **内存容量占用 (Memory Footprint):**
    
    - **常驻内存 (Resident Set Size - RSS):** AI模型权重、操作系统、运行时库等永久占用（或长时间占用）的物理内存。
        
    - **峰值内存 (Peak Memory / High-Water Mark):** 在任务运行过程中（特别是数据预处理和后处理环节）动态申请的内存最高点。峰值内存决定了系统是否会因内存不足（Out of Memory - OOM）而崩溃。
        
- **内存带宽 (Memory Bandwidth):**
    
    - CPU与RAM之间的数据交互（前后处理）。
        
    - BPU与RAM之间的数据交互（输入/输出Tensor的传输）。如果带宽不足，BPU和CPU可能会处于等待数据（Stall）的状态。
        
- **内存泄漏 (Memory Leak):** 对于需要7x24小时运行的嵌入式设备，即使微小的内存泄漏也会最终导致系统崩溃。

## **1.4. 带宽/I/O (Bandwidth / Input/Output)**

数据在系统不同单元间的流动效率至关重要。

- **数据输入/输出带宽 (Data Ingress/Egress):** 例如，从摄像头传感器（MIPI-CSI）、网络（Ethernet）或存储设备（eMMC/SD Card）读取原始数据的速率。
    
- **片上系统总线带宽 (SoC Bus Bandwidth):** 数据在CPU、BPU、DDR RAM之间流动的内部总线带宽。这是最容易被忽视的隐形瓶颈。
    
- **存储I/O (Storage I/O):** 如果模型或数据需要从闪存动态加载，存储设备的读写速度会影响系统的启动时间和运行中的响应延迟。

# **2. 具体的资源分析方案（操作步骤）**

以下是针对上述考量方向，提供的量化分析方法和关键性能指标（KPIs）。通常需要使用开发板SDK提供的性能分析工具（Profiler）和标准的Linux命令行工具。

## **2.1. BPU分析**

- **分析方法:**
    
    1. **[[如何监控RDK-BPU实时利用率？|使用官方Profiler]]:** 运行开发板SDK自带的性能分析工具，在两个AI任务并发运行时进行trace。
        
    2. **API级别计时:** 在代码中，对调用BPU执行推理的API（如 `bpu_run()` 或 `execute()`）前后加入高精度时间戳，计算每次推理的精确耗时。
        
- **关键性能指标 (KPIs):**
    
    - **BPU利用率 (%):** `BPU Busy Time / Total Time`。由Profiler直接提供。目标是了解其饱和程度。
        
    - **模型推理延迟 (ms/inference):** 通过API计时获得。例如，`Model_A_Latency` 和 `Model_B_Latency`。
        
    - **任务吞吐量 (FPS - Frames Per Second):** `1 / (Total Pipeline Latency)`。这是端到端的性能体现。
        
    - **调度间隔时间 (µs):** 通过Profiler观察，上一个BPU任务结束到下一个BPU任务开始之间的时间差。这部分时间即为调度开销或空闲。

## **2.2. CPU分析**

- **分析方法:**
    
    1. **使用 `top` 或 `htop`:** 宏观观察总体CPU利用率、各核心负载及进程排名。
        
    2. **使用 `mpstat -P ALL 1`:** 查看每个CPU核心的实时利用率，定位负载不均问题。
        
    3. **使用 `pidstat -u -p <PID> -t 1`:** 针对性地分析AI任务进程（及其所有线程）的CPU占用，可以区分用户态（User）和内核态（System）的开销。将前后处理逻辑放在独立的线程中有助于更清晰地进行分析。
        
    4. **使用 `perf` 工具 (如果可用):** 进行函数级别的性能剖析，精确定位前后处理代码中的热点函数。
        
- **关键性能指标 (KPIs):**
    
    - **总CPU利用率 (%):** 系统整体繁忙程度。
        
    - **核心#N利用率 (%):** 用于判断是否存在单核瓶颈。
        
    - **前后处理CPU时间 (ms/frame):** 使用 `pidstat` 结合任务的FPS，估算每帧数据处理消耗的CPU时间。
        
    - **内核态CPU占用比例 (%):** 过高的内核态占用通常意味着频繁的I/O操作或驱动交互。
        
    - **上下文切换次数/秒 (cswch/s):** 由 `pidstat -w` 提供。数值异常增高可能表示线程调度过于频繁。

## **2.3. 内存分析**

- **分析方法:**
    
    1. **使用 `free -h`:** 查看系统总内存、已用和可用内存的概况。
        
    2. **使用 `top` 或 `htop`:** 查看各进程的VIRT（虚拟内存）和RSS（物理内存）占用。
        
    3. **读取 `/proc/<PID>/status`:** 查看 `VmHWM` (Peak resident set size)，这是最准确的峰值物理内存占用指标。
        
    4. **使用 `Valgrind (memcheck)`:** 在开发阶段，用Valgrind运行程序以检测内存泄漏（注意：会极大拖慢程序速度，不适用于生产环境分析）。
        
    5. **[[如何监控RDK-DDR带宽占用|SDK内存分析工具]]:** 部分SDK提供工具分析内存带宽占用。
        
- **关键性能指标 (KPIs):**
    
    - **系统常驻内存 (MB):** 系统启动并加载两个任务后的稳定内存占用。
        
    - **任务峰值内存 (MB):** 运行过程中，AI任务进程的 `VmHWM` 值。
        
    - **系统可用内存 (MB):** `Total Memory - System Resident Memory - Task_A_Peak - Task_B_Peak`。
        
    - **[[使用stream测试DDR带宽|内存带宽 (GB/s)]]:** 如果工具支持，记录BPU和CPU访问内存的带宽数据。

## **2.4. 带宽/I/O分析**

- **分析方法:**
    
    1. **使用 `iostat -d -x 1`:** 监控存储设备的读写速率、繁忙程度。
        
    2. **手动计时:** 对数据从输入源到进入预处理函数的时间进行打点计时，反算出实际输入带宽。
        
    3. **SDK Profiler:** 通常会提供BPU与内存之间的数据传输（DMA）时间和速率。
        
- **关键性能指标 (KPIs):**
    
    - **输入数据速率 (MB/s):** 衡量数据采集环节是否是瓶颈。
        
    - **Tensor传输时间 (ms):** 输入Tensor从RAM到BPU，以及输出Tensor从BPU到RAM的时间。

# **3. “后续业务可占用的资源”评估框架**

基于上述分析，按照以下步骤评估未来业务代码的资源可用性。

## **步骤 1: 建立当前系统性能基线 (Baseline Establishment)**

- 在**最大负载**条件下（例如，处理最复杂的输入数据，以最高速率并发运行两个AI任务）持续运行系统一段时间。
    
- 使用第2部分的方法，完整记录所有关键KPIs的**峰值（Peak）**和**平均值（Average）**。
    
    - CPU峰值利用率 (所有核心)
        
    - 内存峰值占用 (整个系统)
        
    - BPU利用率
        
    - 各项I/O带宽

## **步骤 2: 定义资源安全裕度 (Define Safety Margin)**

- 为保证系统稳定性和应对突发负载，切勿将100%的剩余资源分配给新业务。
    
- 根据业务的可靠性要求，设定一个安全裕度，**通常建议为20% - 30%**。

## **步骤 3: 计算可分配的资源预算 (Calculate Available Resource Budget)**

- **CPU预算:**
    
    - **可用CPU百分比:** `(100% - CPU峰值利用率) * (1 - 安全裕度)`
        
    - _结论示例:_ “在当前双AI任务峰值负载下，CPU总体仍有40%的空闲。扣除25%安全裕度后，未来业务代码可长期占用约 **30%** 的CPU资源。如果CPU有4个核心，这相当于大约 **1.2个** 核心的计算力。”
        
- **内存预算:**
    
    - **可用内存大小 (MB):** `(系统总内存 - 系统峰值内存占用) * (1 - 安全裕度)`
        
    - _结论示例:_ “系统总内存1GB，当前业务峰值占用550MB。可用物理内存为450MB。扣除30%安全裕度后，未来业务代码可使用的常驻和动态内存总和不应超过 **315MB**。”
        
- **BPU预算:**
    
    - **结论:** “BPU为专用计算单元，**不可用于**运行非AI的通用业务代码。”
        
    - **评估:** BPU利用率已经高达95%，几乎没有空闲周期。未来任何需要BPU的AI任务都将与现有任务**抢占资源**，导致现有任务的延迟增加或吞吐量下降。未来非AI业务**无可用BPU资源**。
        
- **带宽/I/O预算:**
    
    - 评估方式同CPU，计算各总线/接口的 `(总带宽 - 峰值使用带宽) * (1 - 安全裕度)`。
        
    - _结论示例:_ “内存带宽峰值占用60%，未来业务代码的数据密集型操作需谨慎，避免CPU与BPU争抢内存带宽导致AI任务性能下降。”

## **步骤 4: 迭代开发与验证 (Iterative Development & Validation)**

- 在开发新的业务代码时，应持续使用上述分析工具进行监控。
    
- 每集成一个新功能模块，都应重复步骤1的基线测试，确保各项资源的占用没有超出预算。如果超出，则需要对新代码或原有代码进行性能优化。

通过这套框架，您可以从被动地“感觉系统还有资源”转变为**数据驱动**地、精确地评估和规划系统未来的扩展能力，从而保证整个系统的稳定性与性能。
