# 最佳实践

```Docker
...

CMD ["tail", "-f", "/dev/null"]
```

# 替代方案

```Docker
...

CMD ["/bin/bash"]
```

==**必须配合**== ==`**docekr run -itd**`== ==**创建容器**==

## 工作原理分解

1. `**CMD ["/bin/bash"]**` **在 Dockerfile 中的作用**
    - 这行指令设置容器的默认启动程序为 `/bin/bash`。
    - `bash` 是一个 Shell 程序。当它启动时，会检查自己是否连接到了一个终端（TTY）。
        - **如果没有终端**：它会认为自己是用来执行脚本的，执行完（或发现无事可做）后就会立即退出。
        - **如果连接了终端**：它会进入交互模式，显示一个提示符（比如 `root@<container_id>:/#`），然后等待用户输入命令。在这种模式下，它不会退出。
2. `**docker run -itd**` **命令的作用**
    - `d` (`-detach`)：在后台运行容器。
    - `i` (`-interactive`)：保持标准输入（STDIN）打开，即使没有连接。这是与 `t` 配对使用的。
    - `t` (`-tty`)：**这是最关键的标志**。它为容器**分配一个模拟的虚拟终端（pseudo-TTY）**。

**当两者结合时：**

1. `docker run` 命令启动容器。
2. 因为你使用了 `t` 标志，Docker 为这个容器创建了一个虚拟终端。
3. 容器执行默认命令 `CMD ["/bin/bash"]`。
4. `bash` 进程启动后，检测到自己被连接到了一个 TTY 上。
5. 因此，`bash` 进入了**交互模式**，并保持运行状态，等待输入。
6. 即使容器是在后台运行（因为有 `d`），这个等待输入的 `bash` 进程（作为 PID 1）也**不会退出**。
7. 所以，容器会持续在后台运行，你可以随时使用 `docker exec` 进入它。

**如果缺少** `**-t**` **会怎样？** 

如果你运行 `docker run -id my-image`（没有 `-t`），容器会**立即退出**。因为 `bash` 启动后发现没有连接到终端，它无事可做，于是就退出了，从而导致容器停止。

# 对比上述两种方案

|   |   |   |
|---|---|---|
|特性 (Feature)|`CMD ["/bin/bash"]`|`CMD ["tail", "-f", "/dev/null"]` (更推荐)|
|**运行命令**|`docker run -itd ...`**必须记住加** `**-t**`|`docker run -d ...`**命令更简单**，不需要 `-it`|
|**健壮性**|**脆弱，容易出错**。如果团队中有人不熟悉，忘记加 `-t`，容器就会启动失败，难以排查。|**非常健壮**。`tail` 命令本身就是持续运行的，不依赖任何运行时参数，所以 `docker run -d` 总是能成功。|
|**意图清晰度**|**意图模糊**。只看 Dockerfile，你不知道这个镜像的设计者是希望它交互式运行，还是希望它运行某个脚本。|**意图非常清晰**。看到 `tail -f /dev/null`，任何人都能立刻明白：这个镜像是被设计用来在后台持续运行，以供后续 `exec` 进入的。|
|**资源占用**|几乎无差别，但 `bash` 是一个功能完备的 Shell，比 `tail` 略微复杂。|极其轻量，只做一个工作。|
