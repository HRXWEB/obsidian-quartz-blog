# DDS 在 ROS 生态中扮演的角色

你可以把 DDS 看作 ROS 2 通信的**“底层基础设施”**或**“操作系统”**。它不再是一个需要用户手动配置和启动的独立组件（像 ROS 1 的** `roscore`），而是一个**内置的、去中心化的通信引擎。

它主要扮演了以下几个核心角色：

- **即插即用的网络发现（Discovery）：**
    - ROS 1 依赖 `roscore` 这个中心服务器来注册和查找节点。
    - ROS 2 的 DDS 则实现了**去中心化的节点发现**。当一个 ROS 2 节点启动时，它会通过多播（Multicast）等方式，自动在网络中广播自己的存在，并发现其他正在运行的节点。这就像你把一台电脑插上局域网，它能自动找到其他所有设备一样，无需中央服务器。
- **高效可靠的数据传输（Data Transport）：**
    - DDS 专门为实时和分布式系统设计，它负责处理所有底层的数据传输细节，包括**序列化**（将数据结构转换为字节流）、**网络通信**和**反序列化**。
    - 它的传输效率很高，能够处理像点云、视频流这样的高吞吐量数据。
- **强大的服务质量控制（Quality of Service, QoS）：**
    - 这是 DDS 最强大的特性之一。它提供了丰富的 QoS 策略，让开发者可以根据不同数据流的特性，精细地调整通信行为。
    - 例如，对于传感器数据流（如摄像头、激光雷达），你可以使用**“尽力而为（Best Effort）”**策略，牺牲数据包的可靠性来换取时效性。
    - 而对于重要的控制指令或参数，你可以使用**“可靠（Reliable）”**策略，确保每一条消息都成功送达。
- **ROS 2 的“大脑”：**
    - DDS 不仅仅是 ROS 2 的通信通道，它还定义了 ROS 2 的基本通信模型。ROS 2 中的话题、服务、参数和动作，都是基于 DDS 的发布-订阅（Pub/Sub）和请求-响应（Request/Response）模式实现的。

# 为什么可以有不同的 DDS 存在？

这正是 ROS 2 架构设计的精妙之处，它体现了**“抽象化”和“模块化”**的核心思想。

你可以用一个**汽车**的例子来理解：

- **ROS 2 应用程序**（你的代码）就像是**汽车的驾驶员**。你只需要知道如何踩油门、刹车和转动方向盘。
- **ROS 2 客户端库**（如 `rclcpp`、`rclpy`）就像是**汽车的驾驶舱接口**（油门踏板、方向盘）。它提供一个标准的 API，让你无需关心底层机械细节。
- **ROS 2 中间件接口（RMW）就像是汽车的变速箱**。它是一个标准的 API 层，连接驾驶舱接口和实际的引擎。不管你的车用的是哪家厂商的引擎，它都能通过这个变速箱正常工作。
- **不同的 DDS 实现**（Fast DDS, Cyclone DDS 等）就像是**不同品牌的汽车引擎**。它们都遵守相同的标准（DDS 协议），都实现了 RMW 接口。

这种分层设计带来了巨大的好处：

- **供应商中立性（Vendor Neutrality）：** ROS 2 不再被绑定到某一个特定的通信协议或公司。如果 Fast DDS 出现问题或有更好的替代品出现，开发者可以轻松切换，而无需修改上层应用代码。
- **自由选择的灵活性（Choice & Flexibility）：** 开发者可以根据自己的项目需求，选择最适合的 DDS 实现。
    - 对于你遇到的嵌入式设备，Cyclone DDS 的轻量化和高效性使其成为理想选择。
    - 对于需要高级功能和大规模部署的项目，Fast DDS 提供的额外功能可能更有优势。
- **鼓励竞争与创新（Competition & Innovation）：** 这种开放的架构鼓励 DDS 供应商之间进行竞争和创新，从而不断提升性能、降低内存占用，为 ROS 社区带来更好的底层技术。
- **拥抱行业标准（Adopting Industry Standards）：** ROS 2 选择了 DDS 这个在工业自动化、航空航天等领域广泛应用的行业标准，这使得 ROS 2 更加健壮、可靠，并且能够与其他非 ROS 的 DDS 应用进行互操作。

# RMW 接口主要定义了以下核心概念的抽象

- **节点（Node）：** 定义了节点的创建、销毁和生命周期管理。
- **话题（Topic）：** 抽象了话题的发布者（Publisher）和订阅者（Subscriber），包括创建、销毁、发布消息和接收消息的 API。
- **服务（Service）：** 抽象了服务的服务端（Server）和客户端（Client），定义了服务请求和响应的机制。
- **动作（Action）：** 抽象了动作的服务器和客户端，包括目标的发送、反馈的接收和结果的获取。
- **类型支持（Type Support）：** 这是 RMW 的关键职责之一。它定义了如何将 ROS 2 消息类型（`std_msgs/msg/String` 等）注册到 DDS 中，并提供在不同数据类型之间进行序列化和反序列化的方法。
- **服务质量（QoS）：** RMW 定义了 QoS 策略的接口，确保 `rcl` 传递给 DDS 的 QoS 参数是统一的。
- **等待集（Wait Set）：** 这是一个非常重要的抽象。它允许一个线程同时等待多个通信事件（如多个话题的接收、服务的请求等），这对于编写高效、非阻塞的单线程 ROS 2 节点至关重要。

# 如何切换不同的DDS

以 cyclonedds 为例

```bash
sudo apt install ros-humble-rmw-cyclonedds-cpp -y
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp

# 切换回 fastdds
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
```

# 为什么只需要安装一个包，设置环境变量就可以切换 DDS？

- **运行时选择：** 当一个 ROS 2 应用程序（如一个节点）启动时，它首先需要加载一个 RMW 实现库。
- **查找机制：** ROS 2 的运行时机制会检查 `RMW_IMPLEMENTATION` 环境变量的值（例如 `rmw_cyclonedds_cpp`）。
- **动态链接：** 根据这个环境变量的值，ROS 2 应用程序会**动态加载**对应的共享库。
    - 如果设置为 `rmw_cyclonedds_cpp`，系统就加载 `librmw_cyclonedds_cpp.so`（或其他平台的文件）。
    - 如果设置为 `rmw_fastrtps_cpp`，系统就加载 `librmw_fastrtps_cpp.so`。
- **即时切换：** 所有的 ROS 核心功能指针都指向这个被加载的库中的函数。因此，你只需要在命令行中更改环境变量并重新运行你的 ROS 节点，就可以在不重新编译任何 ROS 节点代码的情况下，**即时切换**底层的 DDS 实现。

# 不同 DDS 的比较

下是从几个关键角度对主要 DDS 实现进行的详细比较：

| 比较角度         | Fast DDS                                                                                    | Cyclone DDS                                                                      | RTI Connext DDS                                                                           | OpenDDS                                                   |
| ------------ | ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| **开发方**      | eProsima                                                                                    | ZettaScale Technology (Eclipse Foundation)                                       | RTI (Real-Time Innovations)                                                               | Object Computing, Inc.                                    |
| **许可模式**     | Apache 2.0 (完全开源)                                                                           | Eclipse Public License 2.0 (完全开源)                                                | 商业许可（提供部分开源版本）                                                                            | Apache 2.0 (完全开源)                                         |
| **设计理念**     | 功能丰富、高度可配置、企业级。致力于成为通用、全面的 DDS 解决方案。                                                        | 轻量、高效、实时、嵌入式友好。专注于低延迟和高吞吐量。                                                      | 高可靠、高安全、高可扩展性。为任务关键型应用而设计。                                                                | 模块化、功能全面。旨在提供一个开放、可扩展的 DDS 平台。                            |
| **性能**       | **吞吐量**: 在处理大数据流时表现出色。**延迟**: 在小数据量和高频率场景下，延迟表现非常优秀。                                        | **延迟**: 通常被认为是低延迟方面的佼佼者，特别是在嵌入式和实时系统中。<br>**内存**：占用极低。                           | **性能**: 业界标杆，特别是在可靠性和可预测性方面。其性能针对特定应用进行了高度优化。                                             | **性能**: 整体表现不错，但在某些极端性能测试中可能不及 Fast DDS 和 Cyclone DDS。    |
| **核心功能**     | **多播/单播**: 支持。<br>**Discovery Server**: 独有功能，用于在复杂网络中简化节点发现。<br>**安全**: 提供 DDS-Security 插件。 | **多播/单播**: 支持。<br>**Discovery Server**: 不原生支持。<br>**安全**: 专注于核心 DDS 功能，安全功能相对基础。 | **发现**: 支持丰富的发现机制。<br>**安全性**: 提供业内最全面的 DDS-Security 实现。<br>**其他**: 丰富的工具链、可视化工具、调试和分析工具。 | **模块化**: 高度模块化，允许用户根据需要选择协议。<br>**安全性**: 支持 DDS-Security。 |
| **ROS 2 集成** | **默认 DDS**，拥有最大的用户群和最完善的文档。                                                                 | 官方支持，在社区中流行度很高。                                                                  | 官方支持，但需要额外安装 RTI 的 SDK。<br>由于其商业属性，使用门槛较高。                                                | 官方支持，但使用率相对较低。                                            |
| **典型应用**     | 通用机器人开发、大型机器人车队、工业自动化系统。                                                                    | 自动驾驶（传感器数据处理）、实时控制、嵌入式机器人系统。                                                     | 国防、航空航天、医疗设备、金融交易等对可靠性和安全性要求极高的领域。                                                        | 需要可定制化和开放性、非实时性要求那么高的应用。                                  |

**总结：**

- **Fast DDS** 是一个很好的通用选择，因为它既有强大的功能，又有活跃的社区支持。对于绝大多数 ROS 2 项目来说，它都是最安全、最方便的默认选项。
- **Cyclone DDS** 是在嵌入式和实时系统中的**专业工具**，如果你对性能和资源有严格要求，它可能是最佳选择。
- **RTI Connext DDS** 是**行业标准**，如果你在做需要最高级别可靠性、可预测性和安全性的任务关键型项目，并且愿意为之支付商业许可费用，那么它是最可靠的选择。
- **OpenDDS** 则是一个功能全面的开源备选项，适用于那些希望完全掌控 DDS 源码并进行高度定制化的项目。
