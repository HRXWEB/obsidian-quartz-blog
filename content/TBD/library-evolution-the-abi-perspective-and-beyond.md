# 一、 引言：为什么版本演进是 SDK 的“终极考验”？

- 库开发者的噩梦：用户升级了 DLL，程序却随机崩溃。
- 核心冲突：快速迭代的需求 vs 极其脆弱的二进制边界。
- 明确定义：API（源代码兼容）与 ABI（二进制兼容）的区别。

# 二、 二进制兼容（ABI）的深水区

- **内存布局的稳定性**：为什么结构体中间加个字段就是“自杀”？
- **虚函数的陷阱**：虚函数表（vtable）的偏移量是如何被破坏的。
- **符号解析（Symbols）**：函数签名改变导致的“找不到入口点”错误。

# 三、 实战：确保 ABI 兼容的工程技巧

- **结构体版本化与 Size 校验**：
    - 利用 `version` 或 `sizeof` 字段作为“内存保命符”。
    - 构造函数自动设置版本的技巧。
- **Pimpl 模式（编译防火墙）**：
    - 核心原理：指针指向实现。
    - 为什么它是 100% 隐藏内部变更的终极方案。
- **保留字段（Padding）**：
    - 预留 `reserved[8]` 字段的艺术：为未来留出余地。

# 四、源代码兼容（API）：编译通过是第一步

虽然本次讨论的核心是 ABI，但 API 兼容性是其前提。它关注的是用户代码在升级 SDK 后，无需修改即可重新编译成功。

- **API 与 ABI 的协同与背离**：
    - 为什么有些改动能通过编译（API 兼容），却会导致运行时崩溃（ABI 不兼容）？
    - 典型案例：增加**默认参数**（API 的福音，ABI 的杀手）。
- **保持函数签名（Signature）的稳定**：
    - 参数类型、数量及顺序的固定化。
    - 为什么严禁在已有接口中修改 `const` 或 `volatile` 属性。
- **防止符号冲突（Name Collisions）**：
    - 为什么 C 风格 SDK 必须坚持使用**唯一前缀**（如 `YueSDK_`）。
    - 避免在头文件中污染全局命名空间。
- **为 ABI 留出余地**：
    - 预留 API 接口的扩展性，以便在不改变签名的情况下演进内部实现。

# 五、 行为兼容性：看不见的“暗箭”

- **语义连贯性**：为什么改了一个排序算法的稳定性也会导致破坏。
- **副作用管理**：不要在原本“纯粹”的函数里偷偷增加 IO 或锁操作。
- **最小惊讶原则**：升级后，老用户期望的结果不应发生突变。

# 六、 版本管理策略：给用户明确的预期

- **语义化版本 (SemVer)**：Major.Minor.Patch 到底该怎么跳号？
- **优雅的废弃（Deprecation）**：使用 `[[deprecated]]` 引导用户平滑迁移，而不是直接暴力删除接口。
- **自动化测试保障**：如何通过“冒烟测试”和“跨版本回归测试”守住底线。

# 七、 总结：稳定胜过一切

- 好的 SDK 是让用户“感觉不到它在升级”。
- 架构设计时的“克制”远比“过度设计”重要。

> **API 兼容是“开发者友好”，确保代码资产的生命周期；** **ABI 兼容是“分发友好”，确保二进制交付的平滑升级。** **作为高级库开发者，我们追求的是：让用户在不重新编译的情况下直接替换 DLL 就能享受新功能，同时在他们决定重新编译时，一行代码都不用改。**

| **修改类型**     | **兼容性质**            | **替换 Lib 即可？** | **需要重新编译？**   |
| ------------ | ------------------- | -------------- | ------------- |
| **仅修改实现逻辑**  | API & ABI 兼容        | **是** ✅        | 否             |
| **增加函数重载**   | API & ABI 兼容        | **是** ✅        | 否             |
| **增加默认参数**   | **API 兼容 / ABI 破坏** | 否 ❌            | **是** ✅       |
| **修改私有成员变量** | **API 兼容 / ABI 破坏** | 否 ❌            | **是** ✅       |
| **重命名头文件**   | **API 破坏 / ABI 兼容** | **是** ✅        | **是** (若需改源码) |
|              |                     |                |               |
