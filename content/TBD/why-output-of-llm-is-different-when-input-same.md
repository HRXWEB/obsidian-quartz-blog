由内到外四个层级来回答：

- 实际输入不同
- 生成算法带来的随机性
- 推理实现不同
- 计算顺序随机性

# 实际输入不同

最典型的比如 RAG，会将实际的输入和检索内容混合后输入到模型，检索的结果有随机性。其余会修改输入的技术还有：

- prompt 压缩
- system prompt 随产品升级而变化
- cache

# 生成算法带来随机性

目前大模型在推理的时候，会有一些参数来控制大模型的输出，比如 **temperature, beam search size**，还有采样策略 topk 和 topp等。甚至 openai 的 api 中就用 seed 这个参数来改变生成策略的随机种子。

temperature 会直接改变输出 token 的概率，而 beam search 和采样策略则是用来避免结果是贪婪搜索生成的。因为贪婪的搜索不一定是全局最优的，甚至会导致文本生成能力退化：输出平淡无奇，缺乏连贯性，还容易[[why-output-of-llm-will-repeat|陷入重复]]。

# 推理实现的不同

- 硬件实现不同：不同型号 GPU 浮点计算不能保证 bit 对齐。大模型的计算量大更加剧了这一问题，误差累积。
- 软件实现不同：vllm、sglang、variant tesnor compute lib

# 计算顺序随机性

> [!EXAMPLE] 性质
> 数值计算中，浮点数计算不满足结合律 (A + B) + C != A + (B + C)

而 GPU 在进行矩阵运算的时候，经常会进行矩阵分解，再加上很多推理框架在计算资源紧张的时候会进行抢占处理，这就导致同样的输入，受推理环境的影响其计算顺序是不一样的。
