1） 并发编程三要素：

[原子性](https://zhida.zhihu.com/search?content_id=168760977&content_type=Article&match_order=1&q=%E5%8E%9F%E5%AD%90%E6%80%A7&zhida_source=entity)：一个或者多个操作要么全部执行成功要么全部执行失败。

[有序性](https://zhida.zhihu.com/search?content_id=168760977&content_type=Article&match_order=1&q=%E6%9C%89%E5%BA%8F%E6%80%A7&zhida_source=entity)：程序执行顺序按照代码顺序先后执行，但是CPU可能会对指令进行重排序。

[可见性](https://zhida.zhihu.com/search?content_id=168760977&content_type=Article&match_order=1&q=%E5%8F%AF%E8%A7%81%E6%80%A7&zhida_source=entity)：当多个线程访问同一个变量时，如果一个线程修改了变量，其他线程立即获取最新的值。

2） 线程的五大状态：

创建状态：当用new操作符创建一个线程的时候。

就绪状态：调用start方法，处于就绪状态的线程并不一定马上就执行run方法，还需要等待CPU的调度。

运行状态：CPU开始调度线程，并开始执行Run方法。

阻塞状态：线程的执行过程中可能因为一些原因进入阻塞状态，比如调用sleep方法，获取尝试得到一个锁等等。、

死亡状态：Run方法执行完或者执行中遇到异常。

3） [悲观锁](https://zhida.zhihu.com/search?content_id=168760977&content_type=Article&match_order=1&q=%E6%82%B2%E8%A7%82%E9%94%81&zhida_source=entity)和[乐观锁](https://zhida.zhihu.com/search?content_id=168760977&content_type=Article&match_order=1&q=%E4%B9%90%E8%A7%82%E9%94%81&zhida_source=entity)

悲观锁：认为一定会有其他线程来改变他，所以每次操作就会加锁，会造成线程阻塞。

乐观锁：认为不会有线程来改变他，每次操作不会加锁，但是如果因为其他线程来改变了值，造成了冲突，会因为冲突而操作失败，但是他又会继续重试，直到成功为止，不会造成线程阻塞。

4） 线程之间的协作

线程之间的协作有：wait、notify、notifyAll等

5） Synchronized关键字（同步锁）

修饰一个代码块：被修饰的代码块称为同步语句块，作用范围是大括号{}括起来的代码，作用的对象是调用这个代码块的对象。

修饰一个方法：被修饰的方法称为同步方法，作用范围是整个方法，作用的对象是调用这个方法的对象。

修饰一个静态的方法：作用范围是整个静态方法，所用的对象是这个类中的所有对象。

修饰一个类：作用范围是synchronized后面括起来的部分，作用的对象是这个类中的所有对象

6） CAS

比较替换，是实现并发应用的一种技术，主要用于不想使线程进入阻塞的一种方式，操作包含三个操作数-内存位置V、预期原值A、新值B，如果内存位置和预期原值相匹配，处理器自动将该位置的值改为新值，否则就不做处理。

CAS三大问题：ABA问题、循环时间长开销大、只能保证一个共享变量的原子操作，即一个CAS只能处理一个。

7） 线程池

如果使用一个线程就创建一个，存在如果并发的线程量大，且每个线程执行一个任务就结束了，这样频繁的创建线程就会给系统带来很大的开销，导致系统效率大大降低，因为频繁创建线程和销毁线程是需要时间的，而线程池可以对线程进行复用，大大减少线程创建和销毁所带来的性能消耗。

mutex 和 condition variable 一定是同时出现配合工作的吗？ 

这是一个很好的问题，答案是：是的，条件变量 (Condition Variable) 几乎总是需要和互斥锁 (Mutex) 配合使用才能正常工作和避免竞态条件 (Race Condition)。

💡 为什么条件变量需要 Mutex？

Mutex（互斥锁）和 Condition Variable（条件变量）各自有着不同的作用，但它们在多线程协作中必须共同发挥作用：

1. 互斥锁 (Mutex) 的作用：保护共享数据
互斥访问 (Mutual Exclusion)： Mutex 的主要目的是确保在任何给定时刻，只有一个线程可以访问和修改被它保护的共享数据/状态（这个状态通常就是线程等待的那个“条件”）。
例子： 如果没有 Mutex，一个线程可能正在检查一个布尔变量 ready（即判断条件），而另一个线程同时修改了这个变量并发送了通知。这会导致一个经典的**“遗失的信号 (Lost Wakeup)”** 问题，即等待的线程可能会永远阻塞。
2. 条件变量 (Condition Variable) 的作用：等待和通知
等待 (Waiting)： Condition Variable 允许一个线程在某个条件不满足时，高效地阻塞（即不占用 CPU 资源）并等待。
通知 (Notifying)： 允许另一个线程在条件被满足时，唤醒一个或所有正在等待的线程。
3. 它们如何协同工作？（原子操作是关键）
条件变量的 wait 操作至关重要，它需要完成两个原子性的操作：
释放 Mutex： 线程释放它之前锁住的 Mutex。
进入等待状态： 线程进入阻塞等待状态。
如果这两个步骤不是原子操作，就会出现竞态条件：
错误的顺序： 线程 A 检查条件不满足，释放 Mutex，然后准备进入等待。如果此时线程 B 改变了条件并发送了 notify，线程 A 将错过这个 notify，然后进入无限期的等待。
wait 函数的原子性保证了线程 A 释放锁和进入等待状态是在一个不可分割的操作中完成的，从而防止了信号丢失。当线程被唤醒后，它还会自动重新获取这个 Mutex。

简而言之：

Mutex 保护了条件本身（即共享状态）不被多个线程同时修改，防止数据竞争。

Condition Variable 提供了线程间通信和高效阻塞/唤醒的机制，避免了线程的忙等待（Busy Waiting）。

总结：

单独使用 Mutex： 可以实现互斥访问，但如果需要等待某个条件，将不得不使用忙等待（在一个循环中不断检查条件），这会极大地浪费 CPU 资源。

单独使用 Condition Variable： 不可能！ 条件变量的设计需要 Mutex 来保护它所依赖的共享状态，并且保证 wait 操作的原子性。

因此，在所有主流的并发编程库中，条件变量总是与互斥锁一起使用。

从第一性原理出发，解释为什么二者需要协同工作。另外分别举例：单独使用mutex、合作使用mutex与cv

这是一个非常好的、深入的问题。从第一性原理 (First Principles) 出发，我们可以将多线程程序的核心需求分解为两个基本问题：数据的正确性和线程的同步/通信。Mutex 和 Condition Variable 分别解决了这两个问题。

🚀 第一性原理：为何必须协同工作？

1. 基本需求 I：数据的正确性 (Mutex)
原理： 当多个线程访问和修改共享资源（例如一个计数器、一个队列、或条件变量所依赖的状态变量）时，如果不加保护，就会发生竞态条件 (Race Condition)，导致数据不一致或程序崩溃。
解决方案： Mutex（互斥锁）通过强制互斥访问来解决这个问题。它确保了在任何时刻，只有一个线程可以进入临界区，从而保证了共享数据的修改是原子性的。
2. 基本需求 II：线程的高效同步 (Condition Variable)
原理： 线程经常需要等待另一个线程完成某项工作或将某个共享状态设置为特定值（即“条件”满足）才能继续执行。如果采用忙等待 (Busy Waiting)（即在一个循环中不断检查条件），将极大地浪费 CPU 资源。
解决方案： Condition Variable（条件变量）提供了阻塞/唤醒机制，允许线程在条件不满足时睡眠，释放 CPU，并在条件满足时被通知后唤醒。
3. 协同工作的关键：原子性操作
核心问题出在如何安全地从检查条件过渡到等待睡眠。
想象一个共享布尔变量 condition_met：
线程 A (等待者)
线程 B (通知者)
4. 锁住 Mutex

5. 检查 condition_met (为 false)

6. 释放 Mutex

7. 锁住 Mutex

8. 设置 condition_met 为 true

9. 发送 Notify 信号

10. 释放 Mutex
11. 线程 A 准备进入睡眠
(信号丢失)
9. 线程 A 永远阻塞

在第 3 步和第 8 步之间，线程 B 有机会完成所有工作并发送通知。由于线程 A 还没有真正开始等待，它错过了这个信号（遗失的信号）。

解决方案：

条件变量的 wait(mutex) 操作被设计成一个原子操作，它执行以下步骤：

原子地释放传入的 Mutex。

原子地阻塞当前线程，并将其加入等待队列。

这种原子性保证了：在释放锁和开始等待之间，通知者线程没有机会抢占执行并发送被错过的信号。

总结： Mutex 保证了对状态变量（条件）的读写是正确的；而 Condition Variable 利用这个 Mutex 提供的保护，安全且高效地实现了从检查状态到等待的转换。

📚 实例分析

1. 场景一：单独使用 Mutex (解决数据竞态)
目标： 保护一个共享计数器，防止多个线程同时修改导致计数错误。
元素
作用
共享资源
int counter = 0;
互斥锁
std::mutex m;

C++

void increment() {

    // 锁住 Mutex，进入临界区

    m.lock();

    
    // 临界区：只有当前线程可以访问和修改 counter
    counter++; 
    
    // 释放 Mutex，离开临界区
    m.unlock();

}

// 缺陷：如果需要等待 counter 达到某个值，必须使用忙等待。

void wait_for_ten() {

    while (true) {

        m.lock();

        if (counter >= 10) {

            m.unlock();

            break; // 条件满足，退出

        }

        m.unlock();

        // ❌ 极大地浪费 CPU 资源 (忙等待/自旋锁的低效版本)

        std::this_thread::sleep_for(std::chrono::milliseconds(10)); 

    }

}

2. 场景二：合作使用 Mutex 与 CV (解决同步和通信)
目标： 实现生产者-消费者模型，生产者生成数据，消费者等待数据可用。
元素
作用
共享资源 (状态)
std::queue<int> data_queue;
互斥锁
std::mutex m; (保护队列)
条件变量
std::condition_variable cv; (通知队列有新数据)

C++

// 消费者线程 (等待者)

void consumer() {

    std::unique_lock<std::mutex> lock(m); 

    // 1. wait() 循环：原子地释放锁并睡眠。被唤醒后，自动重新获取锁。
    // wait() 必须在循环中检查条件，以应对“虚假唤醒”
    cv.wait(lock, []{ return !data_queue.empty(); }); 

    // 2. 条件满足（队列非空），在锁的保护下处理数据
    int data = data_queue.front();
    data_queue.pop();

    // 3. 锁在 lock 离开作用域时自动释放
    // (处理数据...) 

}

// 生产者线程 (通知者)

void producer() {

    int new_data = generate_data();

    // 1. 锁住 Mutex，保护共享队列
    std::lock_guard<std::mutex> lock(m); 
    
    // 2. 在锁的保护下修改共享状态 (条件满足)
    data_queue.push(new_data); 

    // 3. 发送通知，唤醒一个或所有等待中的线程
    cv.notify_one(); 
    
    // 4. 锁在 lock 离开作用域时自动释放

}

从多线程编译的第一性原理出发，除了mutex和cv之外，应该还需要哪些组件，各自起了什么作用

这是一个极具洞察力的好问题。如果我们将“第一性原理”定义为多线程编程为了实现正确性、效率和可预测性所必须解决的基础问题，那么除了 Mutex 和 Condition Variable 之外，确实还需要其他核心组件。

这些组件的必要性来自于现代计算机体系结构（如 CPU 缓存、指令乱序执行）和操作系统调度机制。

以下是从第一性原理出发，除了 Mutex（互斥）和 CV（同步/阻塞）之外所需的关键组件及其作用：

🚀 核心组件和第一性原理作用

组件类别

基础问题 (第一性原理)

解决方案 / 作用

对应概念

I. 内存可见性

内存模型问题：一个线程对共享数据的修改，何时能被其他线程看到？（CPU 缓存和编译器优化导致）

强制刷新或失效 CPU 缓存，并阻止编译器和 CPU 重新排序涉及共享内存的操作。

内存屏障 (Memory Barrier) / 原子操作 (Atomic)

II. 线程局部存储

隔离性问题：如何在保持高效的同时，避免所有变量都成为需要锁保护的共享数据？

为每个线程提供一份变量的私有副本，避免了对这些变量的互斥访问需求。

线程局部存储 (Thread-Local Storage, TLS)

III. 异步/协作

任务管理问题：如何高效地组织和切换任务，尤其是在 I/O 密集型或需要细粒度并发的场景？

提供轻量级的并发单元，允许开发者在无需操作系统介入的情况下，在线程内协作切换任务。

信号量 (Semaphore) / 读写锁 (RW Lock) / 协程 (Coroutine/Fiber)

I. 内存可见性：内存屏障 / 原子操作

第一性原理：可见性 (Visibility)

在现代计算机体系结构中，为了提高性能：

CPU 会使用多级缓存 (Cache)： 线程 A 对一个变量的修改可能只写到了它的 L1 缓存，而没有写回主内存，线程 B 看不到这个修改。

编译器和 CPU 会进行指令乱序执行 (Instruction Reordering)： 即使代码是按顺序写的，CPU 或编译器也可能为了效率调整指令顺序，这可能导致依赖关系的破坏。

概念

作用

内存屏障 (Memory Barrier / Fence)

确保所有在屏障之前的内存操作（读/写）都已完成，并且对所有核心可见，然后才能执行屏障之后的内存操作。它是原子操作和锁的底层实现基础。

原子操作 (Atomic Operations)

提供一种方法，让单个变量的读取、写入或修改（如原子加/减）成为不可中断的操作，并且强制必要的内存屏障，保证了操作的原子性和可见性。

为什么 Mutex 不够？ Mutex 只保证临界区的代码是互斥执行的。如果两个线程不使用锁，而是直接操作共享变量，就需要内存屏障来保证可见性。

II. 线程局部存储 (TLS)

第一性原理：隔离性 (Isolation)

如果所有变量都是共享的，那么并发编程的开销将巨大，因为几乎所有变量的访问都需要加锁。

概念

作用

线程局部存储 (Thread-Local Storage, TLS)

将变量声明为线程本地的。这意味着每个线程都拥有该变量的一个私有副本。因此，线程对该变量的读写操作都是隔离的，无需任何锁保护，大大提高了效率。

例子： 存储事务 ID、日志缓冲区或随机数生成器状态。

III. 异步与协作：信号量 / 读写锁 / 协程

第一性原理：任务管理 (Task Management)

Mutex 和 CV 提供了严格的互斥和条件等待。但在某些场景，需要更灵活或更高性能的同步机制。

概念

作用

信号量 (Semaphore)

是一种计数同步机制，用于控制对一组资源的访问权限。它允许多于一个线程（上限由计数器决定）同时进入临界区，解决了 Mutex 只能一进一出的限制。

读写锁 (Reader-Writer Lock)

允许多个读取者同时访问共享资源，但当有写入者时，必须确保互斥。这在读多写少的场景中极大地提高了并发读取的效率。

协程/纤程 (Coroutine / Fiber)

虽然不是传统意义上的同步原语，但它们解决了任务切换的效率问题。它允许在用户空间进行任务切换，避免了昂贵的内核上下文切换，常用于实现高并发的异步 I/O 模型。

这些组件共同构成了健壮、高效的多线程编程的基础。Mutex 和 CV 解决了“如何正确访问并高效等待共享状态”的问题；而 原子操作/内存屏障 解决了“如何让一个线程的修改对另一个线程可见”的问题。
