# 通过 optimum-cli 将模型导出为 onnx 之后，要怎么样推理呢？不是需要分词器吗？KV Cache 怎么办？

`optimum-cli` **导出的 ONNX 模型只是计算图本身，它不包含分词器（Tokenizer）。**

要进行完整的端到端推理，你必须自己处理分词、模型推理和解码这三个步骤。这通常有两种方法：

1. **手动方法**：使用 `onnxruntime` 库手动加载 ONNX 模型并自己编写推理循环（特别是处理 KV 缓存时）。
2. **推荐方法**：使用 `optimum` 库提供的 `ORTModel` 类，它封装了所有复杂的细节，让你能像使用普通 Hugging Face 模型一样调用 `.generate()` 方法。

---

## **核心概念：推理流程**

无论用哪种方法，一个完整的文本生成推理流程总是包含以下三步：

1. **预处理 (Preprocessing)**：
    - 加载与原始模型匹配的 **Tokenizer**。
    - 使用 Tokenizer 将你的输入文本（如 `"中国的首都是"`）转换成模型可以理解的 `input_ids` 和 `attention_mask`。
2. **模型推理 (Inference)**：
    - 将 `input_ids` 和 `attention_mask` 送入 ONNX 模型。
    - 模型输出 `logits`（对下一个词的预测概率分布）。
    - （对于生成任务）如果使用了 `with-past` 优化，模型还会输出 `past_key_values` (KV 缓存) 供下一步使用。
3. **后处理 (Post-processing)**：
    - 从 `logits` 中选择最可能的下一个 token ID（例如通过 `argmax` 或采样）。
    - 将生成的 token ID 追加到输入序列中。
    - 循环执行第 2 和第 3 步，直到生成结束符或达到最大长度。
    - 最后，使用 Tokenizer 的 `.decode()` 方法将所有生成的 token ID 转换回人类可读的文本。

---

## **方法一：手动使用** `onnxruntime` **(更底层，理解原理)**

这种方法能让你清楚地看到每一步发生了什么，但代码更复杂，尤其是在处理 KV 缓存的循环时。

假设你已经用以下命令导出了一个 `gpt2` 模型：

```bash
# 导出为支持 KV 缓存的 ONNX 模型
optimum-cli export onnx --model gpt2 --task text-generation-with-past ./gpt2_onnx/
```

你的 `gpt2_onnx` 文件夹里现在应该有 `model.onnx` (或 `decoder_model.onnx`)、`config.json` 等文件。

**推理代码示例：**

```python
import numpy as np
import onnxruntime as ort
from transformers import AutoTokenizer

# --- 1. 预处理：加载分词器和 ONNX 模型 ---
model_dir = "./gpt2_onnx/"

# 加载分词器（从原始模型或导出目录加载）
tokenizer = AutoTokenizer.from_pretrained(model_dir)

# 加载 ONNX 模型并创建推理会话
# 你可能需要根据实际文件名修改 'model.onnx'
session = ort.InferenceSession(f"{model_dir}/model.onnx")

# 准备输入文本
prompt = "Hugging Face is a company based in New York City"
inputs = tokenizer(prompt, return_tensors="np") # 注意：onnxruntime 使用 numpy

input_ids = inputs.input_ids
attention_mask = inputs.attention_mask

# --- 2. 推理循环 (处理 KV Cache) ---
# 这是手动实现 .generate() 的简化版
max_length = 30
past_key_values = None # 初始没有 past

for _ in range(max_length):
    # 准备模型输入
    # 第一次输入是完整的 prompt，之后是上一个生成的 token
    model_inputs = {
        "input_ids": input_ids,
        "attention_mask": attention_mask
    }
    
    # 如果有 past_key_values，则加入输入
    if past_key_values is not None:
        # 将 past_key_values 添加到输入字典中
        # 注意：这里的键名需要与 ONNX 模型的输入名完全对应
        # 名字通常是 past_key_values.0, past_key_values.1, ...
        for i, past in enumerate(past_key_values):
            model_inputs[f'past_key_values.{i}'] = past

    # 运行推理
    outputs = session.run(None, model_inputs)

    # 提取 logits 和新的 past_key_values
    # 注意：输出名称也需要与 ONNX 模型对应
    logits = outputs[0]
    # 从第二个输出开始是 past_key_values
    past_key_values = outputs[1:]

    # --- 3. 后处理：选择下一个 token ---
    # 使用 argmax 做贪心搜索，获取最后一个时间步的最高概率 token
    next_token_id = np.argmax(logits[:, -1, :], axis=-1)
    
    # 将新 token 添加到输入中，准备下一次循环
    input_ids = np.array([[next_token_id[0]]]) # 下一次的输入只有一个 token
    
    # 更新 attention_mask
    attention_mask = np.concatenate([attention_mask, np.ones((1, 1))], axis=1)

    # 将生成的 token ID 添加到完整序列中
    all_token_ids = np.concatenate([inputs.input_ids, input_ids], axis=1)
    
    # 如果生成了结束符，则停止
    if next_token_id[0] == tokenizer.eos_token_id:
        break

# --- 解码 ---
generated_text = tokenizer.decode(all_token_ids.flatten(), skip_special_tokens=True)
print(generated_text)
```

**这种方法的挑战：** 你需要手动管理 KV 缓存的传递，并确保输入/输出的字典键名与 ONNX 模型的名称完全匹配，这很繁琐且容易出错。

## **方法二：使用** `optimum.onnxruntime.ORTModelForCausalLM` **(强烈推荐)**

`optimum` 库的美妙之处在于它已经为你封装好了上面所有的复杂逻辑。

**使用前提：**

1. 你已经用 `optimum-cli` 导出了模型。
2. 确保在导出目录下有 `tokenizer` 的配置文件（`tokenizer.json`, `vocab.json` 等）。如果没有，可以从原始模型目录复制，或者在加载时直接使用原始模型的名称。最简单的做法是在导出后，将分词器也保存到该目录：
    
    ```python
    from transformers import AutoTokenizer
    tokenizer = AutoTokenizer.from_pretrained("gpt2")
    tokenizer.save_pretrained("./gpt2_onnx/")
    ```

**推理代码示例：**

```python
from optimum.onnxruntime import ORTModelForCausalLM
from transformers import AutoTokenizer

model_dir = "./gpt2_onnx/"

# --- 1. 加载模型和分词器 ---
# Optimum 会自动加载 ONNX 文件并处理好 ONNX Runtime 会话
model = ORTModelForCausalLM.from_pretrained(model_dir)
tokenizer = AutoTokenizer.from_pretrained(model_dir)

# --- 2 & 3. 推理和后处理 (一步到位) ---
prompt = "Hugging Face is a company based in New York City"
inputs = tokenizer(prompt, return_tensors="pt") # 这里可以用 PyTorch Tensors

# 调用 .generate()，就像使用普通 transformers 模型一样！
# Optimum 在底层为你处理了所有 KV 缓存的循环
outputs = model.generate(**inputs, max_new_tokens=30)

# 解码
generated_text = tokenizer.decode(outputs[0], skip_special_tokens=True)
print(generated_text)
# 预测输出:
# Hugging Face is a company based in New York City. We are a team of engineers, designers, and artists who are passionate about building the future of work. We are a
```

## **总结对比**

|   |   |   |
|---|---|---|
|特性|手动 `onnxruntime`|`optimum.ORTModel` (推荐)|
|**易用性**|低，需要手动管理循环和KV缓存|**高**，接口与 `transformers` 完全一致|
|**代码量**|多，繁琐|**少**，简洁清晰|
|**出错率**|高，输入/输出名称容易写错|**低**，由库保证正确性|
|**控制力**|高，可以完全控制每一步的细节|相对低，但已满足绝大多数需求|
|**适用场景**|学习ONNX底层原理、高度定制化的推理流|**生产环境、快速部署、常规应用**|

**结论：** 除非你有非常特殊的理由需要自己控制每一个细节，否则**始终推荐使用** `optimum.onnxruntime.ORTModelForXxx` **类来进行推理**。它极大地简化了流程，让你能专注于业务逻辑而不是底层的技术实现。
