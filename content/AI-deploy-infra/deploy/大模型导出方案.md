# 调研

## Optimum Exporters

### Optimum 将模型转为 ONNX 的核心原理

首先，需要明确一点：**Optimum 本身并不是一个全新的模型转换器，它更像是一个高度自动化和智能化的“调度器”和“配置管理器”**。它在底层主要依赖于 PyTorch 自带的 ONNX 导出功能 (`torch.onnx.export`)。

其工作原理可以分为以下几个步骤：

**A. 核心机制：执行追踪 (Execution Tracing)**

1. **创建虚拟输入 (Dummy Input)**：转换过程的第一步是创建一个符合模型输入要求的虚拟张量（Tensor）。例如，对于一个文本模型，这可能是一个形状为 `[batch_size, sequence_length]` 的随机整数张量。
2. **模型“空跑”**：ONNX 导出器会将这个虚拟输入“喂”给模型，并像执行常规推理一样，让它流经模型的每一层网络（如 Embedding 层、Attention 层、全连接层等）。
3. **记录计算图 (Computational Graph)**：在“空跑”的过程中，导出器会**追踪并记录**下输入数据经过的所有 PyTorch 运算（op）。这个运算序列就构成了一个计算图。例如，它会记录“输入 -> MatMul -> Add -> Softmax -> 输出”这样的流程。
4. **翻译为 ONNX 格式**：最后，导出器会将这个记录下来的、动态的 PyTorch 计算图，**翻译**成一个静态的、与框架无关的 ONNX 图。每一个 PyTorch 运算都会被映射到 ONNX 定义好的标准算子。

**B. Optimum 的关键价值：自动化与配置**

如果只是简单的模型，开发者可以自己手动调用 `torch.onnx.export`。但对于 Hugging Face Hub 上成千上万的复杂大模型，手动操作几乎是不可能的。Optimum 的价值正在于此：

- **自动化配置**：Optimum 为 Hugging Face 上数百个主流模型架构维护了一套专门的 `OnnxConfig` 配置。这个配置告诉导出器所有关键信息：
    - 模型的输入输出叫什么名字 (`input_ids`, `attention_mask`, `logits` 等)。
    - 如何创建正确的虚拟输入。
    - **最重要的：如何处理动态维度 (Dynamic Axes)**。例如，允许 `batch_size` 和 `sequence_length` 是可变的，而不是固定在追踪时的值。这对于生产环境至关重要。
- **处理复杂性**：它能自动处理模型中的特殊结构，比如 KV Caching、注意力机制的掩码等，确保这些逻辑在转换为 ONNX 后依然能正确工作。
- **统一的 API**：开发者无需关心底层复杂的导出脚本，只需一行代码 `ORTModel.from_pretrained("model-name", from_pt=True)`，Optimum 就会在后台自动完成下载、配置、转换和保存的全过程。

**总结来说，Optimum 的原理是利用 PyTorch 的追踪功能，并将这个过程标准化、自动化，通过为海量模型提供预置的最佳转换配置，极大地降低了模型优化的门槛。**

## VLM 模型导出难题

### **A. 多模态与复杂的预处理**

- **问题**：VLM 的输入不是单一的文本或图像，而是两者的结合。其预处理流程通常包含两部分独立的 Python 代码：
    1. **文本处理**：使用 Tokenizer 将文本字符串转换为 `input_ids`。
    2. **图像处理**：使用 `Pillow` 或 `OpenCV` 等库对图像进行缩放、裁剪、归一化等操作，转换为 `pixel_values`。
- **挑战**：这些复杂的预处理逻辑是**在模型外部用 Python 实现的**，它们本身并不是神经网络的一部分，因此**无法被 ONNX 的追踪机制捕捉和转换**。ONNX 只能转换模型内部的张量运算。这意味着你最终得到的 ONNX 模型只包含了核心计算部分，而端到端的完整流程（从原始图片/文本到最终输出）被割裂了。

### **B. 复杂的模型架构**

- **问题**：VLM 通常是“胶水”模型，它将一个独立的视觉编码器（如 ViT）和一个独立的语言模型（如 Llama）通过一个特定的连接模块（如 Projection Layer）“粘合”在一起。
- **挑战**：这种模块化的、拼接式的架构在导出时非常脆弱。两个模型之间的数据流、数据类型和形状对齐必须极其精确，任何微小的不匹配都可能导致导出失败。Optimum 的自动化配置需要完美地适配这种“跨模型”的复杂数据流，难度远超单一模型。

### **C. 动态控制流与自定义算子**

- **问题**：一些先进的 VLM 可能包含动态的逻辑。例如，模型可能会根据图像的内容，决定生成文本的策略。这种依赖于数据值的**控制流**（if/else a>b then...），是静态的 ONNX 图难以表达的。
- **挑战**：如果模型中使用了一些研究性质的、非标准的自定义运算（Custom Operator），而这些运算在 ONNX 中没有对应的标准算子，那么转换就会直接失败。

### **D. 生态系统和工具链尚在发展**

- **问题**：ONNX 和相关优化工具（如 ONNX Runtime, TensorRT）的生态，其发展重心长期以来都集中在 NLP 和主流 CV 领域。
- **挑战**：因此，针对 VLM 这类新兴复杂模型的最佳实践、调试工具和预置配置都还不够成熟。虽然 Hugging Face 团队正在努力为 LLaVA、IDEFICS 等 VLM 添加 ONNX 支持，但其稳定性和覆盖范围还远不及 BERT 或 GPT 这样的经典模型。

### 1. 问题：LLM 部分的 Attention 和 KV Cache

- **您的理解**：非常正确，这是所有自回归模型（包括纯 LLM 和 VLM）在 ONNX 转换中的核心难题。
- **业界解决方案**：**将 KV Cache 显式地作为模型的输入和输出**。
    - 一个支持 KV Cache 的 ONNX 模型通常有两个使用阶段：
        1. **Prefill/Context Phase (首次运行)**：输入是完整的提示（Prompt），包括图片编码和文本编码。模型的输出是第一个 token 的 logits，以及**整个输入序列的 KV Cache** (`past_key_values`)。
        2. **Decoding/Generation Phase (循环运行)**：输入是上一步生成的单个 token ID，以及上一步输出的 `past_key_values`。模型的输出是下一个 token 的 logits，以及**更新后（拼接了当前步）的 KV Cache** (`present_key_values`)。
    - **相关工作**：
        - **Hugging Face Optimum**: `optimum` 在导出支持 KV Cache 的模型时，会自动处理这种输入输出的重构。其 `OnnxConfig` 中会定义好 `past_key_values` 和 `use_cache` 等参数。
        - **NVIDIA TensorRT-LLM**: 这是目前解决此问题的工业级标杆。它内置了高度优化的 Attention 内核（如 FlashAttention/PagedAttention），并提供了一套完整的方案来构建和运行带有 KV Cache 的 VLM（如 LLaVA、IDEFICS）推理引擎。它把这个问题解决得非常好。

### 2. 问题：视觉和文本模块调用频率不一

- **您的理解**：一针见血。视觉编码器（Vision Encoder）每张图只跑一次，而语言模型（LLM）每个生成词都要跑一次。
- **业界解决方案**：**模型解耦 (Model Decomposition)**。将 VLM 拆分为多个独立的 ONNX 模型，由一个外部的“调度器”(Orchestrator)来控制流程。
    - **Vision Encoder ONNX**: 一个独立的模型，输入是图片，输出是图片特征向量（Image Embeddings）。
    - **LLM Decoder ONNX (with KV Cache)**: 另一个独立的模型，负责自回归生成。
    - **工作流程**:
        1. 应用程序调用 **Vision Encoder ONNX** 一次，得到图片特征。
        2. 将图片特征和用户输入的文本 prompt embedding 拼接起来。
        3. 将拼接后的 embedding 送入 **LLM Decoder ONNX**，执行 Prefill 阶段，得到第一个 token 和初始 KV Cache。
        4. 进入循环，每次将上一步的 token 和 KV Cache 送入 **LLM Decoder ONNX**，执行 Decoding 阶段，直到生成结束符。
    - **相关工作**：这种模式是部署 VLM 的事实标准。所有高性能的 VLM 推理框架，包括 **TensorRT-LLM**和一些公司自研的框架，都采用这种解耦架构。

### 3. 问题：图像处理的动态算子

- **您的理解**：完全正确。图像的缩放、裁剪、填充等操作本质上是数据依赖的控制流，很难放入静态的 ONNX 图。
- **业界解决方案**：**将预处理/后处理保留在模型外部**。
    - ONNX 模型应该专注于核心的、计算密集的神经网络部分。
    - 图像的预处理（如使用 `OpenCV`, `Pillow`）和文本的 Tokenization 应该在推理引擎之外，用 Python 或 C++ 等通用语言完成。
    - 这些库已经高度优化，将它们包含在 ONNX 图中不仅困难，而且收益不大。推理框架（如 ONNX Runtime）的输入应该是已经处理好的、干净的 Tensor。
    - **相关工作**：NVIDIA 的 **DALI (Data Loading Library)** 就是一个专门用于构建高性能预处理流水线的工具，它可以在 GPU 上执行图像处理，然后零拷贝地将数据送入推理引擎。

### 4. 问题：文本长度不固定

- **您的理解**：是的，无论是初始 Prompt 还是持续对话，文本长度都在变化。
- **业界解决方案**：**ONNX 的动态维度 (Dynamic Axes)**。
    - 这是一个已经被完美解决的问题。在使用 `torch.onnx.export` (Optimum 会自动做这件事) 导出模型时，可以指定输入的哪些维度是动态的。
    - 例如，一个典型的文本输入 `input_ids` 的形状是 `(batch_size, sequence_length)`。在导出时，我们会将 `batch_size` 和 `sequence_length` 都声明为动态轴。
    - 这样，ONNX Runtime 就能够接收任意长度和任意批次大小的输入，并为其动态分配所需的内存。
    - **Padding 的作用**：当需要将多个不同长度的请求组合成一个批次（Batch）进行推理以提高吞吐量时，才会使用 Padding，将它们填充到该批次中最长序列的长度。对于单个请求，使用动态维度是最高效的。

## 文本生成、翻译、语音转文字等生成式模型

> [!important] 一般来说现在这类模型都是 Decoder Models，只包含 Transformer 架构的解码器部分。

[huggingface optimum](https://huggingface.co/docs/optimum/exporters/onnx/usage_guides/export_a_model#export-a-model-to-onnx-with-optimumexportersonnx) 对这类模型已经提供了很好的支持力度，直接选用即可。

## 如何在导出 onnx 模型时保留 KV Cache 能力

在 optimum-cli 的使用参数中，对于 `--task` 的描述是：

```Plain
--task TASK           The task to export the model for. If not specified, the task will be auto-inferred based on the model. Available tasks depend on the model, but are among: ['default', 'fill-mask', 'text-generation', 'text2text-generation', 'text-classification', 'token-classification', 'multiple-choice', 'object-detection', 'question-answering', 'image-classification', 'image-segmentation', 'masked-im', 'semantic-segmentation', 'automatic-speech-recognition', 'audio-classification', 'audio-frame-classification', 'automatic-speech-recognition', 'audio-xvector', 'image-to-text', 'zero-shot-object-detection', 'image-to-image', 'inpainting', 'text-to-image']. For decoder models, use `xxx-with-past` to export the model using past key values in the decoder.
```

> **For decoder models, use `xxx-with-past` to export the model using past key values in the decoder.**

即，一般 task 参数不需要指定，会从模型推导。但如果想要在 decoder 过程使用 KV Cache，将任务指定为 `xxx-with-past` 即可。

## 动态 shape 如何操作？

针对 R1 NPU 来说只能推理固定 Shape。这个问题需要 NDK 一起出方案。

当前有的模型（比如 qihoo360/fg-clip）可以选择 padding 到固定大小来缓解这个问题。
